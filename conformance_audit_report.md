## 1. Executive Summary

This Master Conformance Audit Report presents a comprehensive, line-by-line security, mathematical, and architectural compliance audit of the Agency-OS platform. The audit was conducted by executing the full control plane verification test suite, performing a detailed database row-level security (RLS) audit, and synthesizing the independent technical findings from the three Explorer agents.

Overall, the platform exhibits **world-class mathematical precision and architectural discipline** in its core flows:
1. **Deterministic Safety Gates & Statutory Firewall (R1.1)**: 100% compliant. High-risk actions are procedurally gated, and LLMs are strictly confined to intent-parsing.
2. **Cryptographic Ledger & Audit Trails (R1.2)**: 100% compliant. Every state change is hashed and chained in a tamper-evident, verifiable log.
3. **DAG-Based Saga Orchestrator (R2.1)**: 100% compliant. Forward execution and reverse topological compensation are fully operational and verified.
4. **Unified Math Engine & iPOAS (R2.2)**: 100% compliant. Ad-performance scoring is read-only/advisory, and Meridian-based offline MMM calibration is strictly decoupled.

However, the audit has identified **7 critical security and architectural gaps** that must be remediated prior to production release. These gaps range from a plaintext HMAC signature bypass and missing webhook deduplication, to a disaster recovery temporary data leak, an unimplemented sync action causing crashes, and architectural drift in GCP Secret Manager project-level tenant isolation.

## 3. Detailed Audit Findings & Gaps

### Finding 1: Outbox Table Lacks Row-Level Security (RLS)
* **Requirement Violated**: R1.3 (Multi-Tenant RLS Coverage)
* **Severity**: MEDIUM
* **Exact File Path & Line Range**: `control-plane/migrations/versions/0c06f3a7b210_fix_outbox_tenant_id.py` (Lines 21–25)
* **Verbatim Code Snippet**:
  ```python
  def upgrade() -> None:
      """Upgrade schema."""
      # ### commands auto generated by Alembic - please adjust! ###
      op.add_column('outbox', sa.Column('tenant_id', sa.String(length=32), nullable=True))
      # ### end Alembic commands ###
  ```
* **Explanation of Gap**: The `outbox` table is a crucial tenant-scoped transactional queue. Although a migration was added to append a `tenant_id` column to the `outbox` table, the migration failed to enable row-level security (RLS) or create the `tenant_isolation` policy on Postgres. Consequently, the table remains unprotected at the database layer.
* **Concrete Remediation Code & Instructions**:
  Create a new Alembic migration to enable RLS and apply the policy to the `outbox` table:
  ```python
  def upgrade() -> None:
      bind = op.get_bind()
      if bind.dialect.name == "postgresql":
          op.execute("ALTER TABLE outbox ENABLE ROW LEVEL SECURITY;")
          op.execute("ALTER TABLE outbox FORCE ROW LEVEL SECURITY;")
          op.execute("DROP POLICY IF EXISTS tenant_isolation ON outbox;")
          op.execute("""
              CREATE POLICY tenant_isolation ON outbox
                USING (tenant_id = current_setting('app.current_tenant_id', true) OR current_setting('app.current_tenant_id', true) IS NULL);
          """)

  def downgrade() -> None:
      bind = op.get_bind()
      if bind.dialect.name == "postgresql":
          op.execute("DROP POLICY IF EXISTS tenant_isolation ON outbox;")
          op.execute("ALTER TABLE outbox DISABLE ROW LEVEL SECURITY;")
  ```

* **Severity**: HIGH

* **Verbatim Code Snippet**:

### Finding 3: HMAC Signature Validation Bypass in Webhook Router
* **Requirement Violated**: R2.3 (Central Webhook Router / Security)
* **Severity**: CRITICAL
* **Exact File Path & Line Range**: `control-plane/app/main.py` (Lines 1353–1359)
* **Verbatim Code Snippet**:
  ```python
      # In production, secret would be retrieved from Secret Manager using conn.secret_ref.
      # For local/testing, we use secret_ref directly as the secret key.
      secret_key = conn.secret_ref

      # 4. Verify signature
      if not await plugin.verify_signature(raw_body, signature, secret_key):
          raise HTTPException(401, "Webhook signature mismatch")
      ```
* **Explanation of Gap**: The central webhook router validates Shopify signatures using `conn.secret_ref` (the literal string name of the secret, such as `tenant-brand-shopify-secret`) as the HMAC key instead of fetching the actual secret content from Secret Manager. This renders the cryptographic validation mathematically broken in production, or implies that the actual secret is stored in plaintext in the database reference column, violating security best practices.
* **Concrete Remediation Code & Instructions**:
  Refactor the webhook endpoint in `main.py` to resolve the actual secret token from GCP Secret Manager using `SecretManagerClient` before performing signature validation:
  ```python
      # Resolve secret from Secret Manager
      try:
          from app.services.secrets import SecretManagerClient
          secrets_client = SecretManagerClient()
          secret_key = await secrets_client.read_secret(conn.secret_ref)
      except Exception as e:
          logger.error(f"Failed to read webhook secret from Secret Manager: {e}")
          raise HTTPException(500, "Internal secret resolution error")

      if not secret_key:
          raise HTTPException(401, "Webhook secret is empty or unconfigured")
  ```

### Finding 4: Missing Webhook Deduplication
* **Requirement Violated**: R2.3 (Central Webhook Router / Deduplication)
* **Severity**: HIGH
* **Exact File Path & Line Range**: `control-plane/app/main.py` (Lines 1313–1401)
* **Verbatim Code Snippet**:
  In `plugin_webhook` endpoint:
  ```python
          for spec in specs:
              row = await loop.propose(s, spec, actor=f"webhook.{provider}")
  ```
* **Explanation of Gap**: Unlike the WhatsApp webhook router (which uses the `ProcessedWebhookMessage` table to record and deduplicate message IDs), the central `/webhooks/plugins/{provider}` endpoint completely lacks a deduplication check. Duplicate webhooks sent by third-party providers (e.g., due to network retries) will result in duplicate operations being proposed to the kernel.
* **Concrete Remediation Code & Instructions**:
  Extract the unique webhook message ID from the headers and run it through the database deduplication helper before processing:
  ```python
      webhook_id = headers.get("x-webhook-id") or headers.get("x-shopify-webhook-id")
      if webhook_id:
          from app.whatsapp import mark_message_processed
          is_new = await mark_message_processed(webhook_id, s)
          if not is_new:
              return {"status": "ignored", "detail": "duplicate webhook"}
  ```

  ```python
                  OpSpec(
                      tenant_id=tenant_id,
  ```

  In `manage.py`:

* **Explanation of Gap**: The Shopify plugin translates incoming order webhooks into a `manage.shopify.sync_order` action. However, this action is entirely missing from the `ManageAdapter.execute()` method. Consequently, any incoming Shopify order webhook will successfully propose an Op but will fail during execution with an `Unknown action: manage.shopify.sync_order` error, crashing the execution worker thread.
* **Concrete Remediation Code & Instructions**:

  ```python
          elif op.action == "manage.shopify.sync_order":

              from app.models import Order
              order_id = op.params.get("order_id")
              amount_minor = op.params.get("amount_minor")
              stmt = select(Order).where(Order.id == order_id)
              res = await session.execute(stmt)
              existing = res.scalar_one_or_none()
              if not existing:
                  order = Order(
                      id=order_id,
                      tenant_id=op.tenant_id,
                      brand_id=op.brand_id,
                      amount_minor=amount_minor,
                      placed_at=op.params.get("placed_at"),

                  session.add(order)

              return ExecResult(ok=True, detail={"message": f"Order {order_id} synced"})
  ```

### Finding 6: Git Worker Credential Isolation Gap
* **Requirement Violated**: R1.5 (Secrets, Privacy, & Isolation)
* **Severity**: HIGH
* **Exact File Path & Line Range**: `control-plane/app/adapters/build_agent.py` (Lines 11–56)
* **Verbatim Code Snippet**:
  ```python
  class BuildAgentHarness:
      def __init__(self, repo_url: str, branch_name: str, access_token: Optional[str] = None):
          self.repo_url = repo_url
          self.branch_name = branch_name
          self.access_token = access_token
      ...
      def clone_and_checkout(self):
          ...
          cmd = ["git", "clone", "--depth", "1", self.repo_url, self.temp_dir]
  ```
* **Explanation of Gap**: The git worker harness (`BuildAgentHarness`) accepts an optional brand-specific `access_token` in its constructor but completely ignores it during actual git operations (clone, push, commit). Cloning and pushing rely entirely on the host process's ambient SSH credentials, violating the strict brand-level credential isolation requirement.
* **Concrete Remediation Code & Instructions**:
  Modify `BuildAgentHarness` to embed the brand's access token in the git clone and push URLs (if using HTTPS) to guarantee tenant credential isolation:
  ```python

          if self.access_token and self.repo_url.startswith("https://"):
              # Format: https://<token>@github.com/owner/repo.git
              authenticated_url = self.repo_url.replace("https://", f"https://{self.access_token}@")
          else:
              authenticated_url = self.repo_url

  ```

### Finding 7: Architectural Drift in Secret Isolation
* **Requirement Violated**: R1.5 (Secrets, Privacy, & Isolation)
* **Severity**: HIGH
* **Exact File Path & Line Range**: `control-plane/app/services/secrets.py` (Line 14) and all active adapters (`manage.py`, `grow.py`, `presence.py`)
* **Verbatim Code Snippet**:
  In `secrets.py`:
  ```python
  project = project_id or os.getenv("GCP_PROJECT", "aos-control-plane")
  ```
  In `manage.py`, `grow.py`, `presence.py` (initialization):
  ```python
  secrets_client = SecretManagerClient()
  ```
* **Explanation of Gap**: The Master Architecture specifies that brand-scoped third-party credentials must reside in that brand's dedicated GCP project's Secret Manager. However, in the current implementation, the `SecretManagerClient` is instantiated without arguments in all adapters. This causes it to default to the shared control plane GCP project (`aos-control-plane`). Although credentials are separated logically by naming prefixes, they are physically co-located in the same shared GCP project, violating the physical isolation boundary.
* **Concrete Remediation Code & Instructions**:
  Adapters must dynamically resolve the tenant's dedicated GCP project ID from the database and pass it during `SecretManagerClient` instantiation:
  ```python
      # In adapters (e.g. manage.py):
      tenant = await session.get(Tenant, op.tenant_id)
      project_id = tenant.gcp_project if (tenant and tenant.hosting_tier == "dedicated") else None
      secrets_client = SecretManagerClient(project_id=project_id)
  ```

## Conclusion & Recommendations

The Agency-OS platform contains an exceptionally solid, mathematically rigorous foundation. The core transaction lifecycle, DAG saga orchestration, statutory firewall, and tamper-evident audit ledger are fully functional and pass 100% of verification tests. 

To achieve full compliance with the Master Architecture and secure the platform against the identified vulnerabilities, it is highly recommended to prioritize:
1. Fixing the Webhook HMAC signature verification bypass (Finding 3) and adding deduplication (Finding 4).
2. Implementing the missing `manage.shopify.sync_order` action in `ManageAdapter` (Finding 5) to prevent worker crashes.
3. Patching the disaster recovery file leak (Finding 2) and the Git worker credential isolation gap (Finding 6).
4. Dynamically resolving the dedicated GCP projects for secret isolation (Finding 7) and enabling RLS on the `outbox` queue (Finding 1).
